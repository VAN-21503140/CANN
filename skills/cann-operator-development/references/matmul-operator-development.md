## Matrix Model and Formats

The canonical Matmul formula is:

```text
C = A * B + Bias
A: [M, K]
B: [K, N]
Bias: [N]
C: [M, N]
```

Ascend C Matmul commonly uses `half` inputs with `float` output, but the practice also covers int8 input with int32 output.

For Cube-efficient execution, understand the hardware-friendly formats:

- `FRACTAL_NZ` / NZ: used for result-like layouts; logically pads, reshapes, and transposes the last two dimensions so low-level blocks are stored in a Cube-friendly order.
- `FRACTAL_ZZ`: typical A-side layout in L0A, row-major block order outside and inside the fractal.
- `FRACTAL_ZN`: typical B-side layout in L0B, row-major block order outside and column-major inside the fractal.
- `ND`: natural tensor layout used by user-facing input/output. The high-level Matmul API can hide parts of format conversion, but tiling and memory layout still need to match the API contract.

Cube units operate on block-shaped matrix fragments. For `half`, a common hardware mental model is 16x16x16. Data formats and tile sizes must feed the Cube unit with contiguous, reusable blocks.

## Matmul Data Flow

Logical storage positions:

- `A1`, `B1`, `C1`: larger cache-like staging locations.
- `A2`, `B2`, `C2`: smaller per-compute locations near Cube execution.
- `CO1`: Cube output fragment.
- `CO2`: aggregated output.

Typical flows:

- A input: `GM -> A2`, `GM -> A1 -> A2`, or `VECOUT -> A1 -> A2`.
- B input: `GM -> B2`, `GM -> B1 -> B2`, or `VECOUT -> B1 -> B2`.
- Compute: `A2 * B2 -> CO1`.
- Aggregate/output: `CO1 -> CO2 -> GM` or `CO2 -> VECIN`.

The high-level `Matmul` API reduces the developer-facing kernel model to:

1. CopyIn: `SetTensorA`, `SetTensorB`, optional `SetBias`.
2. Compute: `Iterate` or `IterateAll`.
3. CopyOut: `GetTensorC` or direct output through `IterateAll`.
4. Cleanup: `End`.

## Host-Side Matmul Tiling

Matmul tiling splits work across M/N and optionally K:

- Multi-core split:
  - Split A along M into `singleCoreM`.
  - Split B along N into `singleCoreN`.
  - One core computes a `singleCoreM * singleCoreN` C tile.
  - `usedCoreNum = (M / singleCoreM) * (N / singleCoreN)` when dimensions divide cleanly.
- Single-core split:
  - Use `baseM`, `baseN`, `baseK` for the inner loop blocks.
  - Ensure local memory can hold the selected fragments and output block.

Use `TCubeTiling` to pass Cube tiling data to Kernel. The course pattern adds at least:

- `uint64_t localMemSize`: UB/local memory size available to the kernel.
- `TCubeTiling cubeTilingData`: Cube tiling parameters generated by the Matmul tiling API.

Common Host-side API flow:

```cpp
auto platformInfo = platform_ascendc::PlatformAscendC(context->GetPlatformInfo());
uint64_t ubSize = 0;
platformInfo.GetCoreMemSize(platform_ascendc::CoreMemType::UB, ubSize);

matmul_tiling::MultiCoreMatmulTiling tiling(platformInfo);
tiling.SetAType(...);
tiling.SetBType(...);
tiling.SetCType(...);
tiling.SetBiasType(...);
tiling.SetShape(singleM, singleN, singleK);
tiling.SetOrgShape(M, N, Ka, Kb);
tiling.SetFixSplit(baseM, baseN, baseK); // when fixed split is desired
tiling.GetTiling(cubeTilingData);
```

Tiling implementation should:

- Set `blockDim` / used core count consistently with M/N split.
- Fill `TCubeTiling` with the same A/B/C/Bias positions, formats, dtypes, transposition settings, and shape assumptions used by Kernel.
- Keep local-memory constraints explicit:
  - C fragment: `baseM * baseN * sizeof(C_dtype)` must fit target output staging.
  - For fused CV operators, the same fragment often must also fit UB for Vector post-processing.
- Register the tiling data class with `REGISTER_TILING_DATA_CLASS` or the local project equivalent.

## Kernel-Side Matmul API Pattern

Kernel implementation normally defines a class such as `MatmulKernel` with:

- `Init`: parse tiling, calculate this core's offsets, set `GlobalTensor` buffers.
- `CalcOffset`: map `GetBlockIdx()` to M/N block coordinates and compute A/B/C offsets.
- `Process`: call the high-level Matmul API.

Core skeleton:

```cpp
using AType = matmul::MatmulType<matmul::TPosition::GM, CubeFormat::ND, half>;
using BType = matmul::MatmulType<matmul::TPosition::GM, CubeFormat::ND, half>;
using CType = matmul::MatmulType<matmul::TPosition::GM, CubeFormat::ND, float>;
using BiasType = matmul::MatmulType<matmul::TPosition::GM, CubeFormat::ND, float>;

matmul::Matmul<AType, BType, CType, BiasType> matmulObj;
AscendC::TPipe pipe;

extern "C" __global__ __aicore__ void matmul_custom(
    GM_ADDR a, GM_ADDR b, GM_ADDR bias, GM_ADDR c,
    GM_ADDR workspace, GM_ADDR tiling)
{
    GET_TILING_DATA(tilingData, tiling);
    REGIST_MATMUL_OBJ(&pipe, GetSysWorkSpacePtr(), matmulObj, &tilingData.cubeTilingData);
    MatmulKernel op;
    op.Init(a, b, bias, c, &tilingData);
    op.Process();
}
```

Process pattern:

```cpp
matmulObj.SetTensorA(aGlobal);
matmulObj.SetTensorB(bGlobal);
matmulObj.SetBias(biasGlobal);
matmulObj.IterateAll(cGlobal);
matmulObj.End();
```

When using `Iterate` instead of `IterateAll`, retrieve each C tile with `GetTensorC` and copy or post-process it explicitly.

## Offset Calculation

For an M/N split:

- Let `mBlockCount = M / singleCoreM`.
- `mBlockIdx = GetBlockIdx() % mBlockCount`.
- `nBlockIdx = GetBlockIdx() / mBlockCount`.
- A offset commonly starts at `mBlockIdx * singleCoreM * K`.
- B offset commonly starts at `nBlockIdx * singleCoreN`.
- C offset commonly starts at `mBlockIdx * singleCoreM * N + nBlockIdx * singleCoreN`.

Adjust offsets for transposed input, batched Matmul, or non-ND layouts. Do not reuse this formula blindly when format or layout changes.

## ACLNN Test Pattern

Single-operator tests normally:

1. Include `acl/acl.h`, generated `aclnn_<op>.h`, and helper headers.
2. Call `aclnnInit`, `aclrtSetDevice`, `aclrtCreateStream`.
3. Allocate host and device memory for A/B/Bias/C.
4. Copy inputs to device.
5. Call `aclnnMatmulCustomGetWorkspaceSize` or the generated workspace-size API.
6. Allocate workspace if needed.
7. Call `aclnnMatmulCustom` or the generated compute API.
8. `aclrtSynchronizeStream`.
9. Copy output back, compare against golden values.
10. Free device memory, destroy stream, reset device, finalize.

For the fixed training example:

- M = 1024, K = 256, N = 640.
- A = all 1, B = all 2, Bias = all 0.5.
- Expected output element is `K * 2 + 0.5 = 512.5`.

## Practice Specs

Chapter 4 practice variants:

- `MatmulCustom`, formula `C = A * B + Bias`.
- Practice 1: `half` input, `float` output, shape M=1024, N=2048, K=512.
- Chapter test: `int8` input, `int32` output, shape M=1024, K=256, N=640, Bias shape N.

For int8/int32 practice:

- Update MatmulType dtypes.
- Check `TCubeTiling` dtype declarations match Host/Kerner assumptions.
- Verify output accumulator type and Bias type are compatible.

## Common Mistakes

- Binding `REGIST_MATMUL_OBJ` to tiling data that was produced with different dtype/format/shape assumptions.
- Setting `blockDim` inconsistent with `usedCoreNum` or M/N split count.
- Forgetting that `SetDim` and `SetBlockDim` mean different things in separated AIC/AIV execution modes.
- Writing C tiles back as if they were contiguous full rows when only a submatrix is being produced.
- Ignoring local memory constraints when increasing `baseM` or `baseN`.
- Using an ND mental model while the lower-level API expects fractal formats.
